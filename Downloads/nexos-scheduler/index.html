CPU Scheduling:
The Art of Fairness & Speed
Every time you open an app, your OS is quietly making thousands of scheduling decisions per second. NexOS lets you see exactly how that works — and why it matters.
Real-World Analogy
Think of CPU scheduling like managing queues at a hospital. The scheduler is the triage nurse — deciding who gets treated first based on urgency (priority), wait time (fairness), or how quickly they can be helped (burst time). A poor scheduling policy means critical patients wait too long. A good one saves lives — and in computing, saves milliseconds that add up to billions in cost.
Why Does This Matter?
Modern computers run hundreds of processes simultaneously, but most CPUs have only a handful of cores. The scheduler decides which process gets CPU time at each moment. A bad scheduler wastes CPU cycles, makes apps feel sluggish, and in real-time systems, can cause catastrophic failures.
Key Concepts
Burst Time
How long a process needs the CPU. Short bursts = I/O-bound. Long bursts = CPU-bound.
Arrival Time
When a process enters the ready queue. Processes don't all arrive at once in real systems.
Turnaround Time
Finish time minus arrival time. Total time a process spends in the system.
Waiting Time
Time spent in the ready queue, not executing. The cost of being preempted or queued.
Response Time
Time from arrival to first CPU execution. Critical for interactive applications.
Starvation
When a process waits indefinitely because higher-priority processes keep arriving.
Where It's Used
CPU scheduling runs inside Linux, Windows, and macOS kernels. Cloud providers like AWS and Google Cloud use scheduling at massive scale to distribute workloads across data centers. Real-time systems in aircraft and medical devices rely on deadline-aware schedulers where a missed deadline can be fatal.
Scheduling Algorithms
Six fundamental approaches to CPU scheduling — each with different tradeoffs between fairness, efficiency, and response time.
FCFS
First Come, First Served
The simplest approach: processes execute in order of arrival. No preemption — once a process starts, it runs to completion.
PreemptiveNo
StarvationNone
ThroughputMedium
Best forBatch systems
SJF
Shortest Job First
Picks the process with the shortest burst time. Optimal for minimizing average waiting time, but requires knowing burst times in advance.
PreemptiveNo
StarvationPossible
Avg WaitingOptimal
Best forKnown workloads
SRTF
Shortest Remaining Time First
Preemptive version of SJF. If a new process arrives with a shorter remaining time, the current process is preempted.
PreemptiveYes
StarvationPossible
Avg WaitingTheoretical min
Best forMixed burst times
Round Robin
Time-Slice Scheduling
Each process gets a fixed time slice (quantum). After the quantum expires, the process is preempted and the next one runs. Fair and responsive.
PreemptiveYes
StarvationNone
ResponseExcellent
Best forInteractive systems
Priority
Priority-Based Scheduling
Processes are assigned priority values; higher-priority processes run first. Can lead to starvation of low-priority processes.
PreemptiveOptional
StarvationLikely
UrgencyRespected
Best forReal-time systems
MLFQ
Multi-Level Feedback Queue
Multiple queues with different priorities. New processes start at the highest level; if they use their full quantum, they're demoted. Adapts to behavior.
PreemptiveYes
StarvationRare
AdaptabilityExcellent
Best forGeneral purpose OS
Edge Cases & Classic Problems
Load these guided scenarios to observe notorious scheduling pathologies in action. Each scenario reveals a fundamental weakness of certain algorithms.
⚠ The Convoy Effect
One long process holds the CPU while all short processes wait behind it — like a slow truck blocking a highway. Classic FCFS failure mode. Try it with FCFS vs SJF.
💀 Starvation
A low-priority process never gets CPU time because high-priority processes keep arriving. Can be catastrophic in production systems.
⏱ Quantum Sensitivity
What happens to Round Robin when the quantum is too small (excessive context switches) vs too large (degrades to FCFS)? Use the What-If simulator.
📊 Mixed Workload
A realistic mix of I/O-bound and CPU-bound processes. Watch how different algorithms handle the heterogeneity differently.
⚠ Starvation Warning
NexOS automatically detects starvation when a process waits more than 3× its burst time. You'll see a red warning banner appear during simulation. This mirrors real-OS aging mechanisms that boost starved processes' priority over time.
Real World Applications
CPU scheduling isn't just an academic concept — it's running right now inside every device you own.
🐧
Linux CFS
Linux uses the Completely Fair Scheduler — a sophisticated variant of MLFQ that tracks "virtual runtime" to ensure every process gets a fair share of CPU time.
🪟
Windows NTFS
Windows uses multilevel priority queues with 32 priority levels. Foreground apps get a priority boost so your active window always feels snappy.
☁️
Cloud Computing
AWS and Google Cloud schedule billions of virtual machines using scheduling principles. Poor scheduling wastes millions in hardware costs.
✈️
Avionics
Aircraft flight computers use real-time scheduling with hard deadlines. Missing a deadline isn't a slowdown — it's a failure. Priority scheduling with deadlines is critical.
🏥
Medical Devices
Pacemakers, infusion pumps, and surgical robots run RTOS schedulers that guarantee execution within microseconds. Human lives depend on correctness.
📱
Mobile OSes
iOS and Android extend scheduling with power awareness — lower-priority background tasks are throttled to preserve battery, while the UI thread always runs first.
The Bottom Line
Efficient scheduling = fewer servers needed = lower cloud costs. A 10% improvement in CPU utilization at Google's scale saves hundreds of millions of dollars annually. The scheduling algorithms you're simulating here are the foundation of all of that.
